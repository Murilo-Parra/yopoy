# MÓDULO 47.0 — Yopoy Strategic Backend-First Foundation & Risk Elimination

## 1. Documento de Arquitetura Backend-First

O Yopoy utilizará uma arquitetura em camadas (Clean Architecture adaptada) com o objetivo de concentrar as regras de negócio em domínios puros, perfeitamente isolados das tecnologias de interface, facilitando auditoria, escalabilidade e testes corporativos rigorosos.

A estrutura de diretórios do núcleo backend será:
```
domain/
application/
infrastructure/
presentation/
```

### Domain
* **Objetivo:** O coração do sistema. Representa as regras inerentes ao funcionamento de uma empresa física.
* **Componentes:** Entidades, Tipos enumerados centrais, Validações estritas, Regras de negócio puras e Invariantes.
* **Regidezes:** Não possui dependências com bibliotecas externas de frontend (e.g., React), bancos de dados ou APIs rest/GraphQL de infraestrutura. Trabalha de forma autossuficiente e em memória.

### Application
* **Objetivo:** Casos de uso concretos que coordenam o comportamento da aplicação.
* **Componentes:** Orchestradores e transações de negócio como `createSale()`, `registerPayment()`, `linkPaymentToSale()`, `openCashSession()`, `closeCashSession()`, `createExpense()`, `createSmartCaptureDraft()`, `archiveDraft()`.
* **Regidezes:** Executa a ponte controlada entre o Mundo Exterior (`presentation`) e o `domain`, delegando salvamentos na `infrastructure` mediante protocolos estritamente tipados.

### Infrastructure
* **Objetivo:** A camada de adaptação e efeitos colaterais. 
* **Componentes:** Repositórios (em PostgreSQL, Supabase, LocalStorage simulado temporário, etc.), Clients de Storage (S3, GCP), e outras integrações externas futuras.
* **Status neste sprint:** Prepara adapters e injeções de dependência, mas **nenhuma migração perigosa ou autorizações SEFAZ serão executadas**.

### Presentation
* **Objetivo:** Consumir a aplicação sem deter inteligência sistêmica profunda.
* **Componentes:** Interfaces React, Desktop e Mobile shells, Componentes de UI puramente visuais.
* **Regidezes:** É expressamente proibido construir lógicas contundentes de cálculos fiscais, fluxo de comanda ou abatimento inter-financeiro nos arquivos `.tsx`. 


## 2. Documento de MVP Operacional

O escopo definido do MVP Operacional do Yopoy assegura foco e barra expansões complexas prematuras: provar a viabilidade diária para empresas de pequeno ou médio porte descontinuarem uso de pranchetas, cadernos manuscritos ou planilhas isoladas, com segurança sistêmica.

**O que o MVP contempla:**
1. Empresa e usuário (Acessos).
2. Cliente simples.
3. Produto/serviço simples.
4. Venda simples.
5. Item da venda.
6. Pagamento.
7. Vínculo pagamento-venda.
8. Caixa do dia.
9. Despesa.
10. Comprovante/anexo.
11. Livro Caixa.
12. Captura Inteligente gerando rascunho.
13. Rascunho manual editável.
14. Arquivamento (eliminação lógica).
15. Auditoria de eventos.
16. Relatório simples do dia.
17. Pasta do contador simples.
18. Pré-nota apenas como rascunho fiscal (estritamente não-emissível).

**O que está expressamente de FORA do MVP Inicial:**
* Emissão oficial e processamento assíncrono real contra web-services SEFAZ;
* Módulos adiantados de inteligência artificial além da leitura passiva (OCR);
* Regimes de automações complexas transacionais e disparos em massa.


## 3. Mapa de Riscos Técnicos

De modo a fundamentar o recomeço alinhado ao escopo, mitigaremos prontamente os 15 riscos passados listados com diretrizes arquiteturais duradouras:

| Risco Mapeado | Tática de Mitigação |
| :--- | :--- |
| **1. Excesso de módulos visuais antes do núcleo real** | Travar a UI até que as lógicas da camada de `application` existam e sejam testáveis de ponta a ponta sem React. |
| **2. Uso excessivo de mock data (falsa sensação de progresso)** | Substituir Mocks soltos por In-Memory Repositories que atestam lógica de domínio até que a Injeção Supabase/DB fique pronta. |
| **3. Falta de modelo de dados central consolidado** | Criar um mapa explícito de relações de tabelas, onde `company_id` orquestra a posse da entidade. |
| **4. Falta de contratos claros entre front e backend** | Padronizaremos todas as requisições mediante Data Transfer Objects (DTOs) pré-forjados em TS. |
| **5. Falta de persistência real controlada** | Tabelas estruturadas; todas as interações devem prever instâncias relacionais imutáveis. |
| **6. Falta de auditoria estruturada** | Criação imediata da entidade AuditLog e injeção do conceito de eventos geradores. |
| **7. Duplicação de RC entre telas diferentes** | Forçar abstração via repositórios do Casos-de-Uso (ex: `linkPaymentToSale`), isolando de eventos visuais. |
| **8. Dados isolados gerados por módulo** | Normalização exigindo uma raiz global sob `company_id`. |
| **9. Avanço pro fiscal antes do MVP operável** | Status bloqueado. Tratamento de Pré-nota exclusivamente como entidade de Rascunho Estático. |
| **10. Construir ERP grande demais prematuramente** | Congelar novas ramificações. O Foco 100% de esforço passa a ser a robustez transacional dos 5 Fluxos Principais de negócio. |
| **11. Confusão entre termos de Rascunho/Pré-nota/etc.** | Mapa de entidades taxativo e léxico unificado no Domínio. (Rascunho != Venda != Pré-nota). |
| **12. Divergências entre mobile e desktop** | Base PWA/Web-First, partilhando o exato código central de `application` para qualquer front conectável. |
| **13. Integrações reais prematuras** | Interfaces simuladas com contratos definitivos de DTO, proibição atual de inserção de `fetchs` reais em instâncias terceiras. |
| **14. SEFAZ, Mq, etc. ativados cedo demais** | Emissão inativa `activationBlocked = true` gerida pelos Módulos (28-46). |
| **15. Ausência de teste ponta-a-ponta (E2E)** | Exigência da aprovação de Casos de Uso integrados que unifiquem criação, interligação e fechamento cronológico dos dados. |


## 4. Mapa de Entidades do Domínio

A base de integridade das regras da ferramenta residirá perfeitamente sobre estas definições.

| Entidade | Finalidade | Campos Sensíveis Resumidos | Dependências |
| :--- | :--- | :--- | :--- |
| **Company** | Root da Empresa Isolada | `id`, `cnpj`, `name`, `status`, `created_at` | N/A |
| **User** | Acesso ao sistema | `id`, `email`, `status`, `company_id`, `created_at` | Company |
| **Role** | Controle em RBAC | `id`, `name`, `permissions`, `company_id` | Company |
| **Customer** | Cliente faturável ou nominal | `id`, `name`, `tax_id`, `company_id`, `created_at` | Company, *Gera auditoria* |
| **Product** | Estoque básico monetizável | `id`, `name`, `sku`, `price`, `company_id` | Company, *Impacta venda futura* |
| **Service** | Serviço monetizável (sem estoq) | `id`, `name`, `price`, `company_id` | Company |
| **Sale** | Ato de negócio consolidado | `id`, `total_amount`, `status`, `company_id`, `created_by` | Company, User, *Gera pré-nota futura* |
| **SaleItem** | Parcela compradora individual | `id`, `sale_id`, `product/service_id`, `qty`, `value` | Sale |
| **Payment** | Entrada de valor concreta | `id`, `amount`, `method`, `status`, `company_id` | Company, *Auditoria Obrigatória* |
| **PaymentLink** | Joint P/V de quitação (Vínculo) | `id`, `payment_id`, `sale_id`, `assigned_amount` | Payment, Sale |
| **CashSession** | Gestão de giro diário nominal | `id`, `opened_at`, `closed_at`, `initial_bal`, `final_bal` | Company, User |
| **CashMovement** | Ato de entrada/saída do Session | `id`, `session_id`, `type`, `amount`, `notes` | CashSession |
| **Expense** | Despesa operacional bruta | `id`, `amount`, `date`, `category`, `company_id` | Company, *Reflete Livro Caixa* |
| **Attachment** | Imagem/Voucher probatório | `id`, `url`, `type`, `parent_entity_id`, `company_id` | Múltiplas |
| **SmartCaptureDraft**| Leitura preditiva OCR pendente | `id`, `raw_text`, `status`, `company_id` | Company |
| **DraftInvoice** | Pré-nota inerte não emissível | `id`, `sale_id`, `simulated_taxes`, `company_id` | Sale |
| **AccountantPackage** | Extrato compactado pro contador | `id`, `period_start`, `period_end`, `status`, `company_id`| Company |
| **LedgerEntry** | Lançamento em diário perpétuo | `id`, `account`, `debit`, `credit`, `company_id` | Company |
| **AuditLog** | Assinatura cronológica passiva | `id`, `entity_type`, `entity_id`, `action`, `user_id` | Company, Imutável |
| **PendingItem** | Dívida/Aviso sem quitação interlig | `id`, `type`, `reference_id`, `amount`, `company_id` | Company |

**Contrato Genérico e Obrigatório Inter-Entidades Sensíveis (Traceability):**
Toda tabela deve comportar o ciclo atômico de:
`id`, `company_id`, `created_at`, `updated_at`, `created_by`, `updated_by`, `status`. Onde aplicável (ex: Vendas e Rascunhos): `archived_at`, `archived_by`, `deleted_at`, `deleted_by`, `version`.


## 5. Mapa de Fluxos Reais do MVP

Estes são os **únicos** e imaculados corações de testes lógicos no momento:

1. **Venda Dinâmica Básica:** 
   `Criação da Venda` -> `Criação do Pagamento` -> `Caixa Recebe Impacto` -> `Livro Caixa é Alimentado Transacionalmente` -> `Exibe no Relatório do Dia`.
2. **Captação Assistida:** 
   `Foto do Recibo/Comanda` -> `Dita Rascunho (SmartCapture)` -> `Agente Revisor (Revisão)` -> `Efetivação em Venda Real` -> `Vínculo de Pagamento Posterior`.
3. **Gerenciamento Passivo:** 
   `Apontamento de Despesa` -> `Anexação do Comprovante (Attachment)` -> `Livro Caixa Debita` -> `Compilação de Movimento para a Pasta do Contador`.
4. **Resolução Transversal Direta (Avulso):** 
   `Pagamento entra via PIX sem Venda Criada` -> `Cria Status de Pendência no Centro de Resoluções` -> `Operador Efetua Vinculação Manual a posteriori` -> `Log de Auditoria Gera Invariante de Match`.
5. **Venda Pós-Paga:** 
   `Venda Efetivada FIada/Pós Paga (Zero Pagamentos)` -> `Levanta PendingItem` -> `Cliente liquida em Data Futura` -> `Pagamento Baixa Parcial/Total`.


## 6. Mapa de APIs Futuras (Camada de Contratos REST/TrRPC)

Endpoints de entrada devem proteger rigorosamente as interações e assegurar `company_id` oriundo do contexto do *Bearer Token* do Usuário conectado, evitando Path Injection.

| Endpoints Essenciais de Operação | Escopo e Funcionalidade Restrita |
| :--- | :--- |
| `POST /api/sales` | Gera Sale com sub-rotina para abstração de Estoque. |
| `GET /api/sales/:id` | Recupera fatos. Exige posse do `company_id` no JWT. |
| `POST /api/payments` | Levanta fundo pecuniário em formato desacoplado. |
| `POST /api/payments/:paymentId/link-sale/:saleId` | Dispara o `linkPaymentToSale` application use-case. |
| `POST /api/payments/:paymentId/unlink` | Estorna correlação. Auditoria Severa engatilhada. |
| `POST /api/cash-sessions/open` | Abre gaveta digital; sela a anterior caso não encerrada. |
| `POST /api/cash-sessions/close` | Trava mutações diárias do giro local. |
| `POST /api/expenses` | Abate passivos com descritivos. |
| `POST /api/smart-capture/drafts` | Ponto de inserção da IA processada via OCR. |
| `POST /api/draft-invoices` | Prepara Preview Fiscal com `activationBlocked=true`. |
| `GET /api/ledger` | Visualização contábil (read-only em front, mutação apenas sub-hook interna). |
| `GET /api/pending-items` | Recupera Matchs faltantes entre (Vendas Vazias vs Payments sem Pai). |
| `GET /api/audit-logs` | Tenda final. Acesso unicamente por Gerência. | 

O contrato de erros garante que: Financeiros, Fiscais futuros ou Auditorianos recusam Hard-Deletes. (403 Forbidden ao tentar `DELETE` nestes). Retorno de erro sempre tipado (Ex: `HTTP 409 Conflict: Payment already unlinked`).


## 7. Mapa de Persistência e Isolamento Multiempresa (Tenancy)

Regras Pétreas para arquitetura de Banco de Dados Postgre(Supabase) ou simulado: 

1. **CompanyID Bound:** Todo o registro do sistema DEVE portar uma coluna `company_id` (Exceção apenas à base gestora do Sistema "Elparrar Admin Root").
2. **PostgreSQL RLS (Row-Level Security):** Toda Query nativa deve invocar implicitamente `where company_id = auth.jwt() -> 'app_metadata' -> 'company_id'`. (RLS ativo default).
3. **Hard-Delete:** Proibido na imensa maioria do domínio; Uso de lógicas Soft-Delete e Arquivamento (`status = 'ARCHIVED'`).
4. Entidades centrais listadas comporão a espinha neural de *Table Initialization*.


## 8. Mapa de Auditoria

A `Auditoria desde o início` será forçada através de Padrões Unitários no caso-de-uso:
* Ações em objetos financeiros operacionais não acontecem sem emitir um `AuditLog(Action, Type, PreviousState, CurrentState)`.
* Trilha gerará um histórico que constrói credibilidade contra quebras de rascunhos em fraudes internas de colaboradores do usuário final.
* Imutável: Sem rota exposta ao front-end do tipo `PUT /api/audit-logs` ou `DELETE /api/audit-logs`.


## 9. Plano de Migração de Mock para Dado Real

Para limpar a dependência estética fictícia:
1. **Fase 1: Implementar In-Memory DAOs:** Substituir objetos e arrays mockados estáticos (.json mock) por Repositórios Interativos em memória que respeitem CRUD. Isso possibilita a constatação de fluxos sem gastar com DB real.
2. **Fase 2: Interligação Lógica E2E:** Fazer a interface React utilizar funções desacopladas de repositório em suas rotas e ações.
3. **Fase 3: Substituição de Adapter:** Apontar as Injections de Repository (de Memória/Mock) para um SDK real de ORM no Backend. 


## 10. Roadmap Técnico Revisado 

O pipeline restrito do Yopoy (Avançando do Módulo 47 em diante):

1. **Domínio do negócio:** Estabilizar interfaces TypeScript puras.
2. **Modelo de dados:** Instanciação em Prisma/Drizzle Types.
3. **Contratos de API:** Setup do protocolo DTO.
4. **Persistência:** (Somente Infra DB, Sem Sefaz).
5. **Auditoria:** Camada transversal inter-conectada.
6. **Fluxos operacionais reais:** Application Logic E2E (Sem UI conectada ainda).
7. **Frontend conectado ao núcleo:** Interfaces passam a chamar o coração, trocando views fakes por funcionais.
8. **Relatórios:** Views focadas em `Ledger` e `Caixa`.
9. **Segurança Avançada:** Permissões RBAC e tokens restritos.
10. **Fiscal em homologação:** Conduzidos sob domínios inibidos e em `Homologation Scope`.
11. **Produção controlada:** Efetivação do ecossistema SEFAZ com responsabilidade.


## 11. Critérios de GO/NO-GO para a próxima etapa

**NO-GO:** 
* Existência de lógicas vitais de cálculo emutáveis no Front-End React (`.tsx`).
* Rascunhos de Telas Visuais implementados sem entidades lógicas de fundo que a embasem arquiteturalmente.

**GO (Para liberar as próximas execuções de Código):**
* Todas as entidades do Módulo 47 devidamente mapeadas em suas representações Types/Interfces do TS dentro da raiz `domain`.
* Roteamento entre Application e Infrastructure atestados contra vazamento de referências cruzadas que acoplem regras aos botões em tela.
* Concordância completa que o "Fiscal Real Sefaz" descansa isolado até ordem mandatória futura, prevalecendo os pilares de Gestão Administrativa, Tesouraria e Despesas.
