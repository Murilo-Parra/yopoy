## Regras de decisão

- problema textual recorrente vira backlog de microcopy;
- problema de hierarquia ou legibilidade vira backlog visual;
- pedido funcional pequeno e compatível com o MVP vira backlog de produto, sem implementação imediata;
- pedido de autenticação, cadastro, banco, migration ou tenant fica congelado;
- pedido de fiscal real fica fora do MVP;
- confusão entre pré-nota e nota fiscal deve priorizar comunicação antes de qualquer recurso;
- ausência de entendimento da Central Visual deve priorizar narrativa e hierarquia, não novos módulos.

Opinião isolada não autoriza ajuste. Toda recomendação deve preservar a separação entre evidência observada, interpretação e decisão.

## Critérios para autorizar próxima implementação

Uma implementação futura somente pode ser proposta quando todos os critérios abaixo forem atendidos:

- [ ] Há evidência do mesmo problema em pelo menos 3 sessões.
- [ ] O problema está claramente classificado.
- [ ] A solução proposta não reabre infraestrutura.
- [ ] A etapa proposta tem allowlist curta e nominal.
- [ ] A solução não envolve auth, cadastro, banco, migration, RLS, tenant ou fiscal real.
- [ ] A Central Visual permanece a mesa única de organização, sem dashboard paralelo.
- [ ] A validação posterior ao ajuste está definida.

## Itens proibidos de virar ação imediata

- unificar autenticação;
- mexer em cadastro mínimo;
- criar migration;
- mexer em RLS;
- criar emissão fiscal;
- implementar integração com SEFAZ;
- criar certificado A1 operacional;
- reescrever `App.tsx`;
- limpar o legado Elparrar em massa;
- criar dashboard paralelo;
- criar módulos antes de validar a mesa.

Pedidos desse tipo devem ser registrados como fora do escopo ou risco estrutural congelado. Não ampliar a allowlist da rodada para atendê-los.
